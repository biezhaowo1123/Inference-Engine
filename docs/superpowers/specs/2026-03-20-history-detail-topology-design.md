# History Detail And Topology Graph Design

## Goal

Extend the existing pages so each history record can be expanded inline to show:

- full inference details
- the existing step-by-step reasoning
- a single topology graph for the whole inference

The topology graph must distinguish between reasoning nodes and fact nodes. Fact nodes are generated by an additional model call when the inference finishes, not on demand when the user opens history.

## Scope

In scope:

- expand history items inline on the existing history page
- persist graph data with the inference task
- generate graph data after the normal step + summary flow completes
- return graph data from the existing inference detail API
- render a lightweight graph without introducing a heavy frontend framework

Out of scope for this iteration:

- a dedicated detail page
- multiple per-step graphs
- graph editing
- advanced graph layout libraries
- graph search/filter/export

## Current State

The current application already has:

- a history list page at `web/templates/history.html`
- an inference detail API at `GET /api/inference/:id`
- inference steps stored in `InferenceStep`
- summary text stored only in the transient inference response, not in task history

The current history page only lists summary metadata for each task and supports delete/clear actions. It does not expose step details inline.

### Inference Flow

`RunInference` keeps its current major phases:

1. resolve provider
2. create task
3. generate step results
4. generate summary

Then add one new phase:

5. generate graph data from the whole inference result

After summary generation, persist:

- summary text for quick display
- structured summary JSON when available
- graph JSON/status/error

The graph prompt should consume:

- task title/domain/subject/change point
- all generated steps
- summary data when available

The model must return JSON only, containing `nodes` and `edges`.

### Failure Behavior

Graph generation must be non-fatal.

If graph generation fails:

- the task still finishes as `completed`
- `GraphStatus` becomes `failed`
- `GraphError` stores a short error message
- `GraphData` remains empty

If graph generation succeeds:

- `GraphStatus` becomes `completed`
- `GraphData` stores the normalized JSON
- `GraphError` is cleared

This keeps the user-facing inference usable even if the graph extraction step fails.

### API Changes

Reuse the existing detail endpoint:

- `GET /api/inference/:id`

Extend the returned task payload with graph fields:

- `summary`
- `summary_data`
- `graph_data`
- `graph_status`
- `graph_error`

Do not add graph data to `GET /api/history`. The list endpoint stays light and only the detail fetch returns the full payload.

## Frontend Design

### History Card Expansion

`history.html` gains:

- a detail toggle button per item
- an inline expandable detail container
- client-side state for the currently expanded task payload

The page behavior:

- load lightweight history list as it does today
- fetch details lazily only when the user opens a card
- cache fetched detail payload in memory for the current page session

### Detail Contents

The expanded section includes:

1. summary/meta
2. steps

### Graph Renderer

Implement a small renderer in page JavaScript:

- parse `graph_data`
- split nodes by type
- assign positions in deterministic rows/layers
- render edges and nodes in SVG

This renderer should remain self-contained in the history page rather than introducing a new shared frontend system.

## Testing Strategy

Backend tests:

- graph generation success persists graph JSON on the task
- graph generation failure does not fail the inference task
- detail API returns graph fields

Frontend verification:

- history item expand/collapse behavior
- detail fetch wiring
- graph empty state for missing/failed graph data
- graph rendering with representative fixture data

Because the project currently lacks a browser test framework, this iteration can rely on:

- Go tests for backend behavior
- targeted manual verification for the history page interaction

## Implementation Notes

- follow the existing page-extension pattern instead of creating a new route
- keep graph JSON normalized and explicit so future versions can migrate to a dedicated graph table or richer renderer without re-parsing prose
- avoid introducing a graph library unless the manual layout becomes a clear blocker

## Risks

- model-generated graph JSON may be malformed
- graph content quality may vary by provider/model
- large inferences may produce crowded layouts

Mitigations:

- parse/validate graph JSON server-side before saving
- keep the graph step non-fatal
- cap node counts or trim low-signal items in the graph prompt if needed
